# TO-HIGH (위로) — Product Principles

> 이 문서는 제품의 **북극성(North Star)**이다. 모든 에이전트는 작업 착수 전 이 문서를 읽고, 변경이 여기 정의된 가치·불변식과 충돌하지 않는지 검증한다. 충돌이 불가피하면 사용자 확인을 먼저 받는다.

---

## 1. 제품 한 줄 정의

**복잡한 감정을 말로 표현하기 어려운 순간에, 선택지 하나를 누르는 것만으로도 상담이 시작되는 AI 심리 상담 서비스.**

- 전문가 개입 전 단계의 **정서적 응급처치(emotional first aid)**
- 치료(therapy)가 아닌 **대화(conversation)**
- 판단하지 않는 **동반자(companion)**

---

## 2. 도메인 용어집 (Ubiquitous Language)

코드·UI·문서·커뮤니케이션에서 **동일한 용어**를 쓴다. 혼용 금지.

### 2.1 상담의 기본 단위

| 용어 | 영문(코드) | 정의 |
|------|----------|------|
| **공책 / 세션** | `Session` | 한 번의 상담 단위. 시작부터 종료(`active`→`completed`)까지. 사용자 UI에서는 "공책"으로, 코드에서는 `Session`으로 부른다. |
| **권** | — | 공책의 수량 단위. 구독 상품 "3권", "10권"은 월간 추가 세션 수. |
| **내담자** | user | 사용자를 프롬프트·상담 맥락에서 부르는 호칭. 상담 윤리상 "고객"·"유저"보다 "내담자"를 우선한다. |
| **턴 / 대화 턴** | context turn | 내담자의 한 입력 + AI 한 응답 = 1턴. `MAX_CONTEXT_COUNT=200`. |
| **컨텍스트** | `context: string[]` | 세션 내 대화 히스토리. 토큰 경제를 위해 임계 초과 시 rolling summary로 압축. |
| **요약** | `summary` | 세션 종료 시 전체 대화의 요약. 재접속 시 참조. |
| **롤링 요약** | rolling summary | 세션 중 context가 길어질 때 앞부분을 요약으로 대체하는 기법. |

### 2.2 상담의 구조

| 용어 | 영문(코드) | 값 | 정의 |
|------|----------|-----|------|
| **카테고리** | `Category` | `self / future / work / relationship / love / daily / other / direct` | 상담 주제. 내담자가 첫 단계에 선택. `direct`는 자유 입력, `other`는 "말로 표현하기 어려움". |
| **상담사 타입** | `CounselorType` | `T / F / reaction / listening` | 상담사 인격. MBTI T(논리)/F(감정) + 짧은 리액션/순수 경청. |
| **응답 모드** | `ResponseMode` | `comfort / listen / organize / validate / direction / similar` | 6가지 응답 스타일. **2026-05 이후 별도 모달 카드가 아닌 채팅 인라인(`ch-inline-modes`)으로 표시.** 데이터 모델은 동일 유지. `MIN_CONTEXT_FOR_RESPONSE=5` 도달 시 노출. 비로그인 사용자는 5턴 도달 시 `loginWall` phase로 전환되어 모드 노출 차단. |
| **선택지** | options | 8개 버튼 | 각 턴에서 내담자가 누를 수 있는 짧은(15자 이내) 반말 반응 선택지. 마지막(8번째)은 항상 "조언해줘" 또는 "정리해줘". 2026-05 단일 흐름 인라인 패턴. |
| **초기 질문** | `INITIAL_QUESTIONS` | 카테고리별 고정 | 첫 질문은 OpenAI 호출 없이 미리 정의된 질문 사용(토큰/속도 절약). |
| **채팅 phase** | `phase` (frontend) | `selecting / mode / loginWall / chat` | UI 단계. 응답 모드 노출 가능 시점에서 토큰 유무로 `mode`/`loginWall` 분기. |
| **로그인 유도** | login wall | 5턴 게이트 | 비로그인 사용자가 `MIN_CONTEXT_FOR_RESPONSE` 도달 시 dim된 대화 히스토리 + 권유 카드 표시. 컨텍스트 보존된 채로 회원가입 유도. |
| **SSE 스트리밍** | `*Stream` API | generator 기반 | `selectOptionStream` / `setModeStream` / `generateResponseStream`. chunk 단위(`question_chunk`, `options`, `metadata`, `next`) 전송. |

### 2.3 응답 모드 6가지 (본질·허용·금지)

| 모드 | 한글 라벨 | 본질 | 허용 | 금지 |
|------|----------|------|------|------|
| `comfort` | 그냥 위로해줘 | 공감·위로 | 감정 반영, 정서적 수용, 지지 | 해결책, "~해보세요", "긍정적으로 생각해" |
| `listen` | 그냥 들어줘 | 경청·최소 개입 | 짧은 반응, 열린 질문 | 분석, 해석, 조언, 긴 응답 |
| `organize` | 상황 정리해줘 | 사실·감정 분리 | 상황 요약, 핵심 포인트, 구조화 | 판단, 평가, 잘잘못 따지기 |
| `validate` | 내가 이상한 건가? | 감정 타당화 | "자연스러운 반응", 보편성 언급 | 진단, 비교, 부정 |
| `direction` | 뭘 해야 할지 모르겠어 | 작은 실천 1개 | 5분 이내 실행 가능한 행동 1개, "안 해도 괜찮아요" | 거창한 목표, 여러 제안, 인생 조언 |
| `similar` | 나만 이런 건가? | 유사 경험 공유 | 일반화된 사례 | 실제/특정 사례, 과장, PII |

**절대 원칙:** 모드는 본질이 섞이면 안 된다. `comfort`에 해결책이 끼면 `comfort`가 아니게 된다. 프롬프트 수정 시 모드 간 경계가 흐려지지 않는지 검토하는 것이 prompt-engineer의 1순위 책임.

### 2.4 상담사 타입 4가지

| 타입 | 정체성 | 핵심 특징 |
|------|--------|----------|
| `T` | 냉철한 조언가 | 사실·논리 분석, 구체적 해결책, 직설적이지만 존중 |
| `F` | 따스한 공감가 | 감정 깊이 수용, 판단 없이 듣기, 정서적 지지 |
| `reaction` | 리액션 상담가 | 1~2문장 짧은 반응, 여백 주기, 자연스러움 |
| `listening` | 경청 상담가 | "네", "그러셨군요" 수준, 거의 질문 없음, 침묵 허용 |

### 2.5 안전 개념

| 용어 | 영문(코드) | 정의 |
|------|----------|------|
| **위기 감지** | Crisis Detection | 자해·자살 관련 표현 키워드 기반 감지(`crisis-detector.ts`). 레벨 `none/low/medium/high`. |
| **위기 레벨** | `level` | `high`(자살·자해 직접 언급) / `medium`(살기 싫음, 포기, 희망 없음) / `low` / `none`. |
| **전문기관 안내** | — | 위기 감지 시 자살예방상담전화 `1393`·청소년상담 `1388` 등 즉시 노출. |
| **PII** | — | 세션 context 원문, 이메일, OAuth provider ID 등. 로그·Sentry·응답 DTO에서 최소화. |
| **익명 사용자** | `userId === 'anonymous'` | 로그인 없이 상담 가능. 권한 경계 반복 검증 대상. |

### 2.6 운영 개념

| 용어 | 정의 |
|------|------|
| **구독 티어** | `basic`(3권/₩2,900) / `premium`(10권/₩7,900) / 무료(3권). `grandfathered` 사용자는 무제한. |
| **세션 한도** | 무료 3권 + 구독 티어 추가. 초과 시 `SESSION_LIMIT_EXCEEDED`. |
| **토큰 경제** | OpenAI 호출 비용 최적화 전략: 초기 질문 하드코딩, 의미없는 입력 필터, rolling summary. |
| **감정 일기(Diary)** | 상담과 별개의 일일 감정 기록. mood(1~5) + 태그(수면/운동/일/사람들/음주/카페인/날씨/건강) + 메모. |

## 3. 핵심 가치 (Core Values) — 우선순위 순

충돌 시 **위가 우선**한다. 이 순서는 바꾸지 않는다.

### V1. 사용자 안전 (User Safety)

> 위기 신호를 덮는 기능은 어떤 이유로도 머지되지 않는다.

- 위기 감지(high/medium)는 어떤 응답 모드·상담사 타입보다 우선한다.
- "긍정적으로 생각해", "별거 아니에요" 등 **감정 최소화** 표현을 프롬프트가 유도하지 않는다.
- 전문기관 안내는 애니메이션·지연·A/B 테스트 대상이 아니다. 즉시·명확하게 노출된다.
- safety-auditor가 Critical을 제기하면 배포가 차단된다.

### V2. 판단하지 않음 (Non-judgment)

> 우리는 상담사가 아니라 **대화 상대**다. 진단·처방·평가하지 않는다.

- 의료 진단·약물 조언 금지. "우울증이세요"·"약을 드세요" 유도 금지.
- "그건 잘못된 생각이에요" 같은 판단적 표현 금지.
- 사용자의 선택·감정을 평가하지 않는다. `validate` 모드는 감정의 정당성을 인정하는 도구이지, "맞다/틀리다"를 판정하는 도구가 아니다.

### V3. 진입 장벽 최소화 (Low Friction)

> 말로 표현하기 어려운 사람도 시작할 수 있어야 한다.

- 선택지 8개 버튼만으로 상담이 끝까지 가능하다.
- 로그인 없이(익명) 상담 시작 가능.
- 첫 응답은 OpenAI 지연 없이(하드코딩 질문) 즉시 나타난다.
- UI는 과도한 모션 없이, 16px 이상 폰트, 44px 이상 터치 타겟.

### V4. 점진적 개방 (Progressive Disclosure)

> 내담자가 편한 만큼만 말하게 한다.

- 카테고리 → 선택지 → 응답 모드 순으로 컨텍스트가 자연스럽게 축적.
- 민감 정보를 먼저 묻지 않는다.
- 3턴까지는 파악, 3턴부터 조언 허용(`MIN_TURNS_FOR_ADVICE=3`).
- 사용자가 말을 끊어도 (짧은 "ㅇㅇ" 입력 등) 자연스럽게 받아준다.

### V5. 개인정보 보호 (Privacy)

> 상담 내용은 내담자의 것이다.

- 세션 context 원문은 로그·Sentry·에러 메시지에 평문 노출 금지.
- OpenAI 전송 범위는 최소화(rolling summary로 압축).
- 사용자가 세션을 종료하면 재구동 수단을 명확히 제공.
- Admin API가 context에 닿는다면 권한 검증이 엄격해야 한다.

### V6. 모드·인격의 일관성 (Consistency of Voice)

> 응답 모드와 상담사 타입은 제품의 정체성이다. 모호하게 만들지 않는다.

- 모드 본질 유지(§2.3 금지 항목 준수).
- 상담사 타입 유지(T는 T답게, F는 F답게). 혼종 금지.
- 응답 길이 `150-250자`, 선택지 `15자 이내`, 마크다운 제목 금지 등 포맷 규약은 전역 불변.

### V7. 지속 가능성 (Economic Sustainability)

> 토큰 비용이 서비스 생명선을 단축시키지 않게.

- 초기 질문·의미없는 입력은 API 호출 없이 처리.
- context가 길어지면 rolling summary로 압축.
- 프롬프트 수정 시 토큰 증가분 추정·보고.
- 무료 사용자도 3권(세션)까지는 보장.

## 4. 불변식 (Invariants)

**어떤 변경에도 유지되어야 하는 규칙.** 깨뜨리려면 사용자 승인 + 변경 이력 기록.

### I1. 위기 경로
- `crisis-detector` 키워드 레벨 다운그레이드·삭제는 safety-auditor 승인 필요.
- 위기 감지 시 UI는 전문기관 안내를 **첫 프레임**에 노출.
- 전화번호 상수(`1393`, `1388`)는 상수 파일에 집중 관리, 임의 수정 금지.

### I2. 상담 플로우
- 카테고리 → (선택지 응답 반복) → 응답 모드 선택 → AI 응답의 큰 흐름은 유지.
- 선택지 8개 · 15자 이내 · 반말 · 마지막은 조언/정리 선택지 규약 유지.
- 응답 길이 150–250자 유지.

### I3. 응답 모드 본질
- §2.3 표의 "금지" 항목이 어떤 모드 프롬프트에서도 유도되어서는 안 된다.
- 새 모드 추가 시 기존 모드와 경계 문서화 필수.

### I4. 경계면 정합성
- `apps/api/src/controller/*/dto/*.response.ts`와 프론트 소비부의 필드·타입·optional·enum 값이 1:1 일치.
- 응답 DTO 변경 시 qa-validator 교차 검증 필수.

### I5. 레이어 분리 (백엔드)
- Controller는 Repository를 직접 쓰지 않는다. Service 경유.
- `@InjectModel`은 Repository 내부에서만.
- 프롬프트는 `apps/api/src/prompts/`에만 정의. 인라인 금지.

### I6. 익명 사용자 권한
- `userId === 'anonymous'` 분기가 있는 엔드포인트는 리소스 소유·권한 재검증.
- 관리자 API·결제 API는 익명 접근 불가.

### I7. PII 경계
- 세션 context 원문을 로그·Sentry `extra`·응답 DTO에 실어 보내지 않는다.
- 카드번호는 마스킹된 상태로만 저장.
- 이메일·OAuth ID는 응답 DTO에 필요 없으면 포함하지 않는다.

## 5. 에픽 (Epics)

제품을 구성하는 주요 기능 묶음. 새 기능은 에픽 중 하나에 속하거나 새 에픽을 제안한다.

### E1. 상담 대화 엔진 (Counseling Engine)
- **범위:** chat 모듈, prompts, OpenAI 클라이언트, 선택지 생성, 응답 생성, 스트리밍
- **핵심 가치:** V2, V6, V7
- **소유 에이전트:** prompt-engineer, api-architect

### E2. 세션 수명주기 (Session Lifecycle)
- **범위:** 세션 생성·활성·종료·재개, context 관리, rolling summary, session limit
- **핵심 가치:** V3, V5, V7
- **소유 에이전트:** api-architect

### E3. 안전망 (Safety Net)
- **범위:** 위기 감지, 전문기관 안내 UI, 민감 주제 가드레일, 판단적 표현 차단
- **핵심 가치:** V1, V2
- **소유 에이전트:** safety-auditor, prompt-engineer, web-architect

### E4. 내담자 프로필 (User Profile)
- **범위:** 인증(Google/Kakao OAuth), 사용자 프로필, MBTI/상담사 타입 선호, `grandfathered`
- **핵심 가치:** V3, V5
- **소유 에이전트:** api-architect, web-architect

### E5. 구독·결제 (Subscription & Payment)
- **범위:** 토스페이먼츠 통합, 구독 티어(basic/premium), 세션 한도, 결제 상태 전이
- **핵심 가치:** V5, V7 / safety-auditor가 우회 경로 감사
- **소유 에이전트:** api-architect, web-architect

### E6. 감정 일기 (Diary)
- **범위:** 일일 mood(1~5) + 태그 + 메모 기록, 주간 시각화
- **핵심 가치:** V3, V4, V5
- **소유 에이전트:** api-architect, web-architect

### E7. 문의 (Inquiry)
- **범위:** 사용자 문의 접수, 알림(이메일), 관리자 응답 흐름
- **핵심 가치:** V5
- **소유 에이전트:** api-architect, web-architect

### E8. 관리자 (Admin)
- **범위:** 세션·사용자·결제 관측, 통계, 위기 사례 트래킹
- **핵심 가치:** V1, V5 / 권한이 가장 민감
- **소유 에이전트:** api-architect, web-architect, safety-auditor

### E9. 관측성 (Observability)
- **범위:** Sentry 통합, 로깅 정책, 성능·토큰 사용량 추적
- **핵심 가치:** V5, V7
- **소유 에이전트:** api-architect, web-architect

## 6. 안티패턴 (Anti-patterns)

**절대 하지 말아야 할 것들.** 발견 시 즉시 롤백 대상.

### A1. 위기 신호 약화
- "힘들어", "죽고 싶어" 등 위기 키워드를 "오탐이 많다"는 이유로 감지에서 제외.
- 위기 응답 앞에 긴 애니메이션·로딩을 두어 지연.
- A/B 테스트로 위기 배너 노출 비율 조정.

### A2. 판단·진단 유도
- "우울증일 수 있어요", "전문가 상담이 필요해요"를 AI 응답으로 생성(전문기관 "안내"는 OK, AI 진단은 NOT OK).
- "그 생각은 왜곡됐어요" 류 판단 표현.

### A3. 모드 혼종
- `comfort` 모드에서 해결책 제시.
- `listen` 모드에서 분석·조언.
- `direction` 모드에서 여러 제안 나열.

### A4. 경계면 가정
- 백엔드 응답 확인 없이 "아마 이런 필드가 있을 것"으로 프론트 타입 추측.
- 에러 응답 shape 없이 `response.data.*`를 그대로 사용.

### A5. PII 노출
- `console.log(session.context)` 류 디버깅 코드 커밋.
- Sentry `setExtra`에 사용자 원문 전달.
- 에러 응답에 DB 예외 원문 포함.

### A6. 익명 사용자 권한 누락
- `userId`만 체크하고 `anonymous` 분기를 누락한 엔드포인트.
- Admin 경로에 `JwtGuard`만 있고 role 체크 누락.

### A7. 토큰 낭비
- 첫 질문을 OpenAI로 매번 생성.
- context 전체를 매번 OpenAI에 전달(요약 미적용).
- 프롬프트에 불필요한 반복 지시 추가.

### A8. 용어 혼용
- 같은 개념을 "세션"·"공책"·"상담"으로 섞어 쓰기(UI 용어 정책: 사용자 대상 = "공책", 개발자 대상 = "세션").
- "User"·"사용자"·"내담자"를 맥락 없이 혼용.

## 7. 의사결정 프로토콜

에이전트가 판단이 어려울 때 다음 순서를 따른다.

1. **불변식(§4)에 위배되는가?** → Yes면 즉시 중단, 사용자 확인.
2. **핵심 가치(§3) 간 충돌이 있는가?** → 우선순위 상위 가치를 따른다. V1 > V2 > … > V7.
3. **용어(§2)가 정의되어 있는가?** → 정의된 용어 사용. 없으면 사용자에게 정의 요청.
4. **에픽(§5) 중 어디에 속하는가?** → 소유 에이전트에 위임.
5. **안티패턴(§6)에 해당하는가?** → 대안 모색.
